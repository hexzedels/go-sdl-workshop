# Сломай, чтобы починить — Челленджи

Перед вами Go-микросервис: API для работы с документами, стандартная библиотека, всё компилируется, всё запускается. Одна проблема — внутри **10 уязвимостей**. Каждая выглядит как обычный код, который мог написать любой из нас. Ваша задача — найти их все и починить.

## Быстрый старт (3 шага)

1. **Склонировать репозиторий:**
   ```bash
   git clone https://github.com/hexzedels/go-sdl-workshop
   cd go-sdl-workshop
   ```
2. **Проверить окружение:**
   ```bash
   make verify-setup
   ```
3. **Открыть CTFd и зарегистрироваться:** [https://golangconf.coderat.space](https://golangconf.coderat.space) — создайте команду из 3–5 человек, туда будете сдавать флаги.

## Требования

- Go 1.22+
- Docker (Compose v2+) — нужен в Раунде 4, для Раундов 1–3 достаточно `go run ./cmd`
- `curl`, `jq`
- Рабочий интернет: `go mod download` и CTFd

## Как устроен воркшоп

4 раунда, каждый про свой класс SDL-инструментов:

| Раунд | Тема                            | Инструменты                                                     | Обязательная | Бонусные | Время (хард-кап) |
| ---------- | ----------------------------------- | -------------------------------------------------------------------------- | :----------------------: | :--------------: | :--------------------------: |
| 1          | Статический анализ | SAST                                                                       |            1            |        2        |          20 мин          |
| 2          | Цепочка поставок     | SCA                                                                        |            1            |        2        |          20 мин          |
| 3          | Рантайм                      | Dynamic/Runtime                                                            |            1            |        2        |          20 мин          |
| 4          | Контейнеры                | Container Scanning                                                         |            1            |        0        |          18 мин          |
|            |                                     | **Вступление + настройка**                        |                          |                  |          10 мин          |
|            |                                     | **3 флеш-разбора между раундами**            |                          |                  |           6 мин           |
|            |                                     | **Итоги + SDL-фреймворк + 30-дневный план** |                          |                  |          12 мин          |
|            |                                     | **Буфер / Q&A**                                                 |                          |                  |          14 мин          |
|            |                                     | **Всего**                                                       |       **4**       |   **6**   |     **120 мин**     |

> Каждая уязвимость в CTFd — это **2 челленджа** (эксплойт + фикс), по своему флагу в каждом. Всего 20 челленджей.

### Обязательные vs бонусные

В каждом раунде есть **одна обязательная** уязвимость — её хватит, чтобы разобраться с темой раунда. Когда сдали оба её флага, в CTFd открываются **бонусные**. Баллы: обязательная — 50 + 100 за два челленджа, бонусная — 100 + 200.

### Жёсткие таймеры

На экране — таймер на каждый раунд с 5- и 1-минутными предупреждениями. Когда раунд кончается, начинается флеш-разбор (2 мин). Команды, не успевшие, продолжают тихо — флаги засчитываются до начала следующего раунда.

## Что делать в каждом раунде

1. **Ломаете** — эксплуатируете уязвимость через `curl` или браузер и получаете **флаг эксплуатации** (как именно — см. ниже и в описании челленджа в CTFd).
2. **Чините** — минимальный патч, каждый на ~3 минуты.
3. **Получаете флаг исправления** — `./scripts/verify-fix/<id>.sh` (например, `./scripts/verify-fix/3.1.sh`). Скрипт прогонит тест и, если фикс рабочий, напечатает флаг.
4. **Сдаёте оба флага в CTFd** — каждый в свой челлендж (на уязвимость заведено два: под эксплойт и под фикс).
5. **Если осталось время** — берёте бонусные челленджи (открываются в CTFd автоматически).
6. **Перед следующим раундом** — `make verify-roundN` как агрегированная проверка, что ничего не сломали.

### Где берётся флаг эксплуатации

| Тип челленджа                         | Как получить флаг                                                                                                                                                                                                                                                                                                                                   |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Прямые** (1.x, 2.1, 2.2, 4.x)       | Флаг `GOSDL{...}` приходит прямо в HTTP-ответе вашей атаки — сдаёте в CTFd.                                                                                                                                                                                                                                         |
| **Косвенные** (2.3, 3.1, 3.2, 3.3) | Эффект атаки наблюдается косвенно (спайк pprof, фатал в логах, тайминг). Запускаете `./scripts/verify-exploit/<id>.sh` — скрипт подтверждает эксплойт и печатает флаг. Точные аргументы — в описании челленджа в CTFd.                                  |

Примеры для Раунда 3:

```bash
# 3.1 Goroutine leak — запустить watcher, потом в другом терминале лить нагрузку
./scripts/verify-exploit/3.1.sh

# 3.2 Race condition — watcher читает docker compose logs, нужен `make docker`
./scripts/verify-exploit/3.2.sh

# 3.3 Timing attack — восстановили ключ, передали скрипту
./scripts/verify-exploit/3.3.sh <recovered-api-key>
```

make run

## FAQ

**Порт 8080 занят.** `lsof -i :8080`, убить процесс или поменять порт в `config.yaml`.

**`go mod download` медленный или падает.** Конференционный Wi-Fi нестабилен. Попробуйте `GOPROXY=direct go mod download` или подождите и повторите.

**Docker не запускается.** Для Раундов 1–3 Docker не нужен — достаточно `make run`. Docker потребуется только в Раунде 4.

**Как сдать флаг в CTFd?** На каждую уязвимость в CTFd заведено **два отдельных челленджа** — по одному флагу в каждом:

- **Эксплойт:** для **прямых** уязвимостей (1.x, 2.1, 2.2, 4.x) флаг `GOSDL{...}` приходит в HTTP-ответе вашей атаки. Для **косвенных** (2.3, 3.1, 3.2, 3.3) запустите `./scripts/verify-exploit/<id>.sh` — точные аргументы указаны в описании челленджа в CTFd. Скрипт подтверждает эффект и печатает флаг.
- **Фикс:** всегда через `./scripts/verify-fix/<id>.sh` после правки кода — скрипт гоняет нужный Go-тест и печатает флаг, если фикс прошёл.
- Флаги сдаются в **два разных челленджа** в CTFd, не в одно поле.

**`scripts/verify-exploit/<id>.sh` пишет «No spike / Window elapsed».** Watcher слушает 10 секунд — эксплойт надо запускать **сразу** после старта скрипта и поддать нагрузку. Для 3.1 — 5 батчей по 50 нероутируемых URL (`192.0.2.x:9999`) на `POST /api/webhooks/notify`. Для 3.2 — десятки параллельных логинов (`xargs -P`, `ab`, `&` в bash).

**`scripts/verify-exploit/3.2.sh` падает с «service is not running».** Он читает логи Docker-контейнера. Поднимите сервис: `make docker` (по умолчанию профиль `app`), либо для Postgres: `docker compose --profile postgres up app-pg` + `SERVICE=app-pg ./scripts/verify-exploit/3.2.sh`.

**`scripts/verify-fix/<id>.sh` падает после фикса.** Проверьте, что нет старого запущенного сервера, который держит порт или базу. `make verify-roundN` (например, `make verify-round1`) — агрегированная проверка всех тестов раунда, полезно перед переходом к следующему.

**Сломал базу данных.** `make clean && make run` — пересоздаст `workshop.db`.

**SQLite: "database is locked".** Один процесс на `workshop.db` за раз. Убедитесь, что не запущено несколько инстансов сервера.

**Race condition не воспроизводится.** Соберите сервер с гонкой (`go run -race ./cmd`) и отправьте десятки параллельных запросов на `/api/admin/sessions`. Watcher `scripts/verify-exploit/3.2.sh` поймает фатал в Docker-логах автоматически.

**Нет Docker в Раунде 4.** Скажите фасилитатору — сканирование образа покажут на проекторе, флаг сдаёте как обычно.

**Сервер не стартует: `config.yaml not found`.** Запускайте из корня репозитория (`cd go-sdl-workshop` перед `make run`).
